DOWNLOAD JAVASCRIPT (inline IIFE at bottom of download.html)
------------------------------------------------------------
Constants:
  REPO = 'afaraha8403/kalam'
  RELEASES_API = 'https://api.github.com/repos/' + REPO + '/releases?per_page=10'

Main functions (conceptual):
  detectOSSync() / resolveMacArch() / detectOS()
    → { os: 'windows'|'mac'|'linux'|null, arch: varies }
    Mac arch from userAgentData.getHighEntropyValues(['architecture']) when available;
    default Apple Silicon if unknown on Mac.

  macAssetMatchesArch(nameLower, arch)
  assetMatchesNewPattern(nameLower, platform, arch)
    Matches filenames like -darwin-aarch64, _windows_x64, -linux-amd64, etc.

  pickAssetForOS(assets, osInfo)
    Picks installer by extension order:
      Windows: .msi, .exe
      Mac: .pkg, .dmg, .app.tar.gz  (download page includes .exe in win list)
      Linux: .AppImage, .deb
    Disambiguates by arch when multiple candidates.

  formatBytes(n), fileKindLabel(filename)
  triggerDownload(url) — creates temporary <a>, click(), remove.

  showOsGuide(os), setOsChip(osInfo), setStatusDone()

  run() on DOMContentLoaded:
    1) detectOS → showOsGuide, setOsChip
    2) fetch RELEASES_API (Accept: application/vnd.github.v3+json)
    3) Scan releases: first non-draft stable, first prerelease beta
    4) primaryRelease = stable || beta; hasBoth = stable && beta
    5) Render #all-assets: header + links for primary release assets; if hasBoth, second
       section for beta assets (pulse-colored borders)
    6) pickAssetForOS(primary assets, osInfo) → auto-download after delay (1500ms, or 0
       if prefers-reduced-motion)
    7) #manual-dl href = picked URL or release page; optional beta CTA paragraph with
       flask icon linking to beta asset for same OS
    8) On failure: status message, manual link to releases/latest, showOsGuide(null)

How site downloads connect to the repo (read this when changing either side)
--------------------------------------------------------------------------------
- The website does not host binaries. It calls the GitHub REST API; assets are whatever
  files are attached to GitHub Releases for `afaraha8403/kalam`.
- Filenames must stay consistent with `pickAssetForOS` / `assetMatchesNewPattern` in

  renames bundles or drops an extension (e.g. only NSIS .exe, no .msi), update the site
  logic or users may fall through to the wrong file or the release page.

================================================================================
UNDERLYING CODE THAT ENABLES DOWNLOADS (REPO, NOT docs/)
================================================================================
This is the pipeline that builds installers and attaches them to GitHub Releases — the
same assets `download.html` lists and auto-downloads.

--------------------------------------------------------------------------------
1) GitHub Actions — `.github/workflows/release.yml`
--------------------------------------------------------------------------------
Trigger:
  `on.push.tags: ['v*']` — cutting a version tag starts the pipeline.

Jobs (high level):
  A) `test` (Ubuntu)
     - Installs Linux Tauri deps, Node 22, Rust stable.
     - `npm install` → `npm run check` → `npm run test:unit` → `npm run build`
     - `cargo fmt --check`, `cargo clippy`, `cargo test` in `src-tauri/`.
     - Ensures the tree is green before any release artifacts are built.

  B) `changelog`
     - Reads `CHANGELOG.md` and extracts the section whose heading matches the tag
       (e.g. `## [v0.1.0]` or `## [0.1.0]`) via `awk`.
     - Output is passed as `releaseBody` for the GitHub release. If no section matches,
       body defaults to: "See the assets to download this version."

  C) `build` (needs: test, changelog)
     - `if: startsWith(github.ref, 'refs/tags/v')`, `permissions.contents: write`.
     - Matrix (four rows, `fail-fast: false`):
         * macos-latest + `args: '--target aarch64-apple-darwin'`  (macos_arch: aarch64)
         * macos-latest + `args: '--target x86_64-apple-darwin'`   (macos_arch: x86_64)
         * ubuntu-22.04 + `args: ''`                               (Linux bundles)
         * windows-latest + `args: '--bundles nsis'`               (Windows NSIS)
     - Each row runs `tauri-apps/tauri-action@v0` with:
         * `tagName: ${{ github.ref_name }}`
         * `releaseName: 'Kalam ${{ github.ref_name }}'`
         * `releaseBody: ${{ needs.changelog.outputs.body }}`
         * `releaseDraft: true` — release stays draft until published manually.
         * `prerelease: ${{ contains(github.ref_name, 'beta') || contains(github.ref_name, 'rc') }}`
         * `assetNamePattern: '[name]-[version]-[platform]-[arch][ext]'`
           → produces names the download page matches with substrings like
             `-darwin-aarch64`, `-windows-x64`, `-linux-amd64`, etc.
         * `args: ${{ matrix.args }}` — forwarded to `tauri build`.
         * Optional secrets: `TAURI_SIGNING_PRIVATE_KEY`,
           `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` (for signed updates / bundles).
     - macOS-only follow-up (each mac matrix job):
         * Verifies bundled `Kalam.app` has `NSMicrophoneUsageDescription` in Info.plist.
         * Builds an extra **.pkg** with `pkgbuild` (component = `.app`, install location
           `/Applications`, identifier `com.kalam.voice`, scripts from
           `src-tauri/scripts/macos-postinstall.sh` copied to `pkg_scripts/postinstall`).
         * Output filename: `kalam_${{ github.ref_name }}_macos_${{ matrix.macos_arch }}.pkg`
         * Uploads that file to the same release: `gh release upload "${{ github.ref_name }}" ... --clobber`.
         → The site prefers `.pkg` for Mac in extension order, so these uploads matter.

--------------------------------------------------------------------------------
2) Tauri app config — `src-tauri/tauri.conf.json` (bundle + updater)
--------------------------------------------------------------------------------
- `build.beforeBuildCommand`: `npm run build` (Vite frontend into `dist/`).
- `bundle.active: true`, `bundle.targets: "all"` — local `tauri build` produces the
  platform’s default set (on CI, matrix `args` narrow Windows to NSIS only).
- `bundle.createUpdaterArtifacts: true` — updater metadata/files for the Tauri updater.
- `plugins.updater.endpoints`: 
    `https://github.com/afaraha8403/kalam/releases/latest/download/latest.json`
  (in-app updates, separate from the marketing download page).
- Windows: `bundle.windows.nsis` (installMode both, `nsis-hooks.nsh`).
- Linux: `bundle.linux.deb.postRemoveScript`: `scripts/postrm.sh`.
- macOS: `Info.plist`, `Entitlements.plist`, minimum macOS 11.0.

--------------------------------------------------------------------------------
3) Local / manual builds — `package.json`
--------------------------------------------------------------------------------
- `npm run build` — Vite production build (required before Tauri bundles).
- `npm run tauri:build` / `tauri:build:unsigned` — wrap `tauri build` on a developer
  machine; artifacts land under `src-tauri/target/...` per Tauri’s layout.

--------------------------------------------------------------------------------
4) End-to-end data flow
--------------------------------------------------------------------------------
  git tag v… pushed
       → release.yml test + changelog
       → tauri-action builds per OS/arch, uploads named assets + updater artifacts
       → mac job adds kalam_<tag>_macos_<aarch64|x86_64>.pkg
  Maintainer publishes the draft release on GitHub
       → GET /repos/afaraha8403/kalam/releases appears in download.html
       → Browser picks asset URL → `triggerDownload(browser_download_url)`

No server-side download API in this project: GitHub hosts files and rate-limits API
calls; the download page is static HTML + client-side fetch.
